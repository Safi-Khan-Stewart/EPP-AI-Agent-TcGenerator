# Test Case Generator — Instructions & Workflow

## Overview

This tool automatically generates comprehensive test cases for any Azure DevOps User Story based on its **Title**, **Description**, and **Acceptance Criteria**. It presents a review page in the browser where you can select which test cases to push into ADO.

---

## Usage

```powershell
python test/ado_advisor.py <USER_STORY_ID>
```

**Example:**

```powershell
python test/ado_advisor.py 432288
```

---

## Step-by-Step Workflow

### 1. Fetch User Story Details

The script connects to Azure DevOps and retrieves:

- **Title** — used to understand the feature context
- **Description** — used to understand the full scope and intent
- **Acceptance Criteria** — parsed to extract individual ACs (supports `AC1:`, `AC2:`, `GIVEN/WHEN/THEN` format)
- **State**, **Assigned To**, **Area Path**, **Iteration Path**

### 2. Generate Test Cases

Test cases are generated by analyzing the acceptance criteria and the user story context, strictly following the rules defined in **[Rules.md](Rules.md)**.

> **All test case generation rules (structure, coverage, functional validation, tagging, etc.) are maintained exclusively in `Rules.md`. Refer to that file for the complete rule set.**

### 3. Review in Browser

A local server starts at `http://localhost:8765/review` and opens in your browser.

The review page displays:

- **User Story details** — ID, title, state, assignee, description
- **Acceptance Criteria** — parsed and numbered
- **Generated Test Cases** — each as a card with a **checkbox** (all checked by default)
- **Select All / Deselect All** buttons
- **Counter** — shows how many test cases are selected
- **"Add to ADO" button** — pushes selected test cases to Azure DevOps

### 4. Select & Push to ADO

1. **Review** each generated test case
2. **Uncheck** any you don't want to add
3. Click the green **"Add to ADO"** button
4. The selected test cases are:
   - Created as **Test Case** work items in ADO
   - Steps are written in **ADO's native TCM XML format** (readable in Test Plans)
   - Each test case is **linked to the User Story** with a "Tests / Tested By" relation
5. A **results popup** shows which test cases were created successfully with their ADO TC IDs

### 5. Stop the Server

Press `Ctrl+C` in the terminal to stop the local review server when done.

---

## Test Case Format in ADO

Test cases are written to ADO in the native `Microsoft.VSTS.TCM.Steps` XML format, with proper step numbering and expected results per step. Every AI-generated test case is tagged with **`Gen-AI`** as per `Rules.md` (RULE-FUNC-07).

---

## Linking

Each created test case is linked to the User Story using the relationship:

- **Test Case → User Story**: `Microsoft.VSTS.Common.TestedBy-Reverse` (Tests)

This makes the test case appear under the "Tests" tab of the User Story in ADO.

## Output Files

| File                              | Description                                    |
| --------------------------------- | ---------------------------------------------- |
| `review_test_cases_US<ID>.html`   | Review page served by the local server          |
| `test_cases_US<ID>.html`          | Standalone HTML report (also generated)         |

---

## Acceptance Criteria Parsing

The parser handles `AC1:`/`AC2:` blocks, `GIVEN/WHEN/THEN` clauses, numbered lines, bullet points, and plain text. Coverage and mapping rules are defined in **[Rules.md](Rules.md)**.

---

## Configuration

The following variables must be set at the top of `ado_advisor.py`:

| Variable      | Description                        |
| ------------- | ---------------------------------- |
| `ADO_ORG`     | Azure DevOps organization name     |
| `ADO_PAT`     | Personal Access Token              |
| `ADO_PROJECT` | Target project name                |

## Troubleshooting

| Issue                        | Solution                                                          |
| ---------------------------- | ----------------------------------------------------------------- |
| SSL certificate error        | Already handled — SSL verification is disabled for corporate envs |
| `pip` not found              | Run `python -m ensurepip --upgrade`                               |
| No ACs found                 | Ensure the user story has `Microsoft.VSTS.Common.AcceptanceCriteria` filled in |
| 400 error creating TC        | Check PAT permissions — needs "Read & Write" on Work Items        |
| Port 8765 already in use     | Stop any previous server instance (Ctrl+C) and retry              |
